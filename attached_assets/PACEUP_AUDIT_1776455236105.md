# PaceUp — User-Flow & Edge-Case Audit

Scope: solo workout flow, event hosting, event join/start/participation, tracking & completion, discover/filter/list views. Findings are ordered by flow, then by severity. Each finding is written so you can paste it directly into a Replit agent prompt — file path, line range, what's wrong, how a user hits it, and a fix hint.

Severity:
- **CRIT** — crash, data loss, silent data corruption, or security exposure
- **HIGH** — user-visible incorrect behavior, stuck state, or fails on common path
- **MED** — wrong in edge cases but recoverable
- **LOW** — cosmetic / rare

## 1) Solo workout flow (`app/run-tracking.tsx`, `contexts/LiveTrackingContext.tsx`, save path)

### CRIT — `announcePace()` can divide by zero and feed Infinity to TTS
- File: `app/run-tracking.tsx:587-595`
- What: `const paceMinTotal = totalSeconds / 60 / distance;` — if a caller passes `distance === 0` (e.g. a misconfigured coach interval, the first announcement before any distance accumulates, or a resume where `lastAnnouncedMileRef` equals current distance), `paceMinTotal` is `Infinity` or `NaN`. `pickCoachPhrase` then receives a garbage pace, and `lastAnnouncedPaceRef.current` is set to Infinity (corrupting the next comparison).
- Repro: Set coach interval to 0.5 mi, start a run, trigger an announcement at exactly the threshold boundary where `lastAnnouncedMileRef.current` matches `totalDistRef.current` (also happens on very short activities where `distance` passed in is `< 0.001`). Also reachable by the dev flow that calls `announcePace` without a GPS fix yet.
- Fix:
  ```ts
  const announcePace = useCallback(async (distance: number, totalSeconds: number) => {
    if (Platform.OS === "web") return;
    if (!distance || distance < 0.01 || !Number.isFinite(distance)) return;
    if (!Number.isFinite(totalSeconds) || totalSeconds <= 0) return;
    const paceMinTotal = totalSeconds / 60 / distance;
    if (!Number.isFinite(paceMinTotal) || paceMinTotal <= 0) return;
    // ... rest unchanged
  ```

### HIGH — `saveRun()` uses stale `elapsed` state instead of `elapsedRef.current`
- File: `app/run-tracking.tsx:1347-1360`
- What: `const pace = distance > 0.01 ? (elapsed / 60) / distance : null;` and `durationSeconds: elapsed` both read the React state `elapsed`, not the ref. If the user taps Finish between timer ticks, or React batches the last `setElapsed`, the saved duration/pace is off by up to 1 second, and PR tier math on the server is computed against the wrong pace.
- Repro: Finish a run at exactly the tick boundary (easy to reproduce in devtools). Saved `durationSeconds` is 1s less than what the user saw on screen.
- Fix: replace both `elapsed` references on lines 1349 and 1357 with `elapsedRef.current`.

### HIGH — Distance clamp to `0.001` still lets zero-distance runs be saved
- File: `app/run-tracking.tsx:1355`
- What: `distanceMiles: Math.max(distance, 0.001)` means any accidental start/stop (tapping Start then Finish without moving) still writes a row. That row pollutes PR math (`server/storage.ts` `getSoloRunPrTiers`), `totalMiles`, `milesThisMonth`, and the streak logic.
- Repro: Tap Start, immediately tap Finish, confirm save. A 0.001 mi run is persisted. Multiple of these skew your "longest run" and "fastest pace" rankings.
- Fix: require a minimum meaningful distance AND duration to save. Either show "Too short to save" and discard, or gate in `saveRun`:
  ```ts
  if (distance < 0.05 || elapsedRef.current < 30) {
    Alert.alert("Too short to save", "Record at least ~50 m and 30 s.");
    setSaving(false);
    return;
  }
  ```

### HIGH — Resume after pause drops the last coordinate → first post-resume fix can inflate distance
- File: `app/run-tracking.tsx:1192-1197`
- What: `handleResume` nulls `lastCoordRef.current` and `lastCoordTimestampRef.current`. The very next `handleCoord` has no previous point, so it falls into the "first fix" branch and silently skips the leg. That's fine if the user stayed put, but if the user **moved** during pause (common: walked into a coffee shop then resumed), none of that distance is counted. Worse, the pause-guard logic that should reject huge jumps after resume is now disabled because there's no reference coord.
- Repro: Run 1 mi → Pause → walk 200 m to a coffee shop → Resume → GPS locks onto new spot. Distance should still read ~1.00 mi, but there's no visual cue that the 200 m was dropped. If you invert the test (stay put and resume), sometimes the GPS has drifted 30-50 m during pause and the drift is also correctly dropped — but the existing accuracy filter already handles that.
- Fix: Keep `lastCoordRef` but add a "resumedAt" flag; on the first fix after resume, require the accuracy filter to pass AND the segment distance `< 50 m` before counting it. That way users who actually moved while paused get a small jump instead of losing the whole leg silently, while teleports from resume-in-car are still rejected.

### HIGH — Recovery after crash restores `prevMileElapsedRef` from total duration, not last mile boundary
- File: `app/run-tracking.tsx:441-442` (the crash-recovery autosave restore block)
- What: On restore, `prevMileElapsedRef.current` is seeded with the full `data.durationSeconds` instead of the elapsed time at the last completed mile split. The *next* mile split computes `segmentSeconds = elapsedRef.current - prevMileElapsedRef.current`, which will be a tiny number (or negative) and get clamped, showing a bogus split like `1:00 /mi`.
- Repro: Force-quit the app at 2.3 mi, 15:00. Reopen — tracking resumes. Cross the 3 mi mark — split chart shows a ~1:00/mi split that wasn't earned.
- Fix: Either persist `prevMileElapsedRef`/`lastSplitMileRef` in the autosave payload, or rehydrate from `data.splits`:
  ```ts
  const completedSplits = (data.splits ?? []).filter((s: any) => !s.isPartial);
  lastSplitMileRef.current = completedSplits.length;
  // Reconstruct cumulative time through the last full mile
  prevMileElapsedRef.current = completedSplits.reduce(
    (acc: number, s: any) => acc + s.paceMinPerMile * 60, 0
  );
  ```

### MED — Double-tap Start spawns two location subscriptions
- File: `app/run-tracking.tsx:1143-1150` (`handleStart`)
- What: No guard on `phase`. A fast double-tap or a re-render during the countdown-kickoff can call `watchLocation()` twice; both callbacks then feed `handleCoord` and double-count distance until one of them is torn down.
- Fix: top of `handleStart`, add `if (phase !== "idle") return;`.

### MED — Tiny partial-mile split labels render as "0.3" on chart
- File: `app/run-tracking.tsx:1229-1237`
- What: On finish, a partial-mile split with label `"0.3"` is pushed to `mileSplitsRef` when the user finishes between mile boundaries. The `MileSplitsChart` component plots it alongside `"M1" / "M2"` etc. with no visual distinction that it's partial, and for very short runs (< 0.1 mi) the pace is clamped to the 1-30 range and produces nonsense.
- Fix: only push partial split when `remainingDist >= 0.1`; otherwise drop. In the chart component, dim / hatch partial bars.

### MED — Pace shows `0:00 /mi` for ultra-short runs instead of `—`
- File: `app/run-tracking.tsx:1349` and the done-screen pace renderer (same file, search `paceM`, `Math.floor(displayPace)`)
- What: `pace` is null for `distance <= 0.01`, but the renderer does `Math.floor(displayPace) || 0`, so null becomes `0:00`, implying a Usain-Bolt-grade pace.
- Fix: render `—` when pace is `null`/`undefined`/`!isFinite`. Unify the display threshold to match the save gate (MED above).

### MED — `lastCoordRef` not cleared when countdown actually fires "GO"
- File: `app/run-tracking.tsx:862-870` (in `handleCoord`) and the countdown→active transition
- What: During the 3-2-1 countdown, GPS callbacks are still running and `lastCoordRef` is set. When the transition to `active` happens, the first active-phase fix uses that stale coord to compute a segment — so any drift during the countdown is counted as real distance.
- Fix: in the effect that transitions `phase: "countdown" → "active"`, null `lastCoordRef` and `lastCoordTimestampRef`, same as what `handleResume` does.

### MED — Offline save has no persistent outbox
- File: `app/run-tracking.tsx` `saveRun` (line ~1347) + `MAX_RETRIES = 3`
- What: The retry loop only retries in-memory. If the user force-kills the app or loses the network for >a minute, the finished run is gone forever. `AUTOSAVE_KEY` is removed in the success path — there's no offline-queue file the app retries on next launch.
- Fix: on final retry failure, keep the autosave row (don't `removeItem`) and add a "Unsynced activities" indicator plus a retry on app resume. At minimum, surface a modal: "Couldn't upload — we saved it locally, try again later."

### MED — Permission-revoked-mid-run not gracefully handled
- File: `app/run-tracking.tsx` (watch subscription) + `contexts/LiveTrackingContext.tsx`
- What: If the user revokes Location while the app is backgrounded, the watcher errors out but the timer keeps ticking and the UI keeps showing the old distance. User comes back, finishes, saves — getting a run with no new GPS data for the last N minutes.
- Fix: subscribe to permission-change events (on Android, poll `Location.getForegroundPermissionsAsync()` on foreground) and either auto-pause or show a persistent banner "Location off — distance paused."

### LOW — Coach announcements skip when `interval` is `NaN`
- File: `app/run-tracking.tsx:889`
- What: `parseFloat(coachIntervalRef.current)` — if the user's stored preference is malformed (migration from older version, empty string), `interval` is `NaN` and the `>=` check is always false, silently disabling the coach. No user feedback.
- Fix: `const interval = parseFloat(coachIntervalRef.current); if (!Number.isFinite(interval) || interval <= 0) { /* fall back to 1 mi */ }`.

## 2) Event hosting (`app/create-run/index.tsx`, `app/run/[id].tsx`, server)

### CRIT — Event edit corrupts scheduled time across timezones
- File: `app/run/[id].tsx:464-476` (the date/time edit parser)
- What: MM/DD/YY parsed as a `Date` and then `.toISOString()`'d. `Date` constructed with "06/15/26 7:30 AM" is local, but `toISOString()` converts to UTC — if the server later formats this in UTC for display (it does, see next), the event time shifts by the user's UTC offset on every edit.
- Repro: PST user opens an event, edits time to "7:30 AM", saves. Reopen the event — shows "3:30 PM" because server formatted the UTC value directly.
- Fix: either always store and send ISO/UTC and format exclusively in user's local tz on render, or store the original wall-clock string + tz name. Stop mixing. At minimum: when editing, construct `new Date(y, m-1, d, hh, mm)` locally and send `.toISOString()` — then on render always use `toLocaleString(undefined, { /* local */ })`, never `timeZone: 'UTC'`.

### CRIT — Creation allows past dates
- File: `app/create-run/index.tsx:244-261` and `server/routes.ts` POST `/api/runs` (around 1150-1165)
- What: Neither client nor server rejects `dateTime <= now`. The late-start-monitor then auto-deletes the "orphan" 60 minutes later, silently. Any participants who join vanish along with the event.
- Fix (client): after you compute `dateTime`, add `if (dateTime.getTime() < Date.now() - 60_000) { alert("Event must be in the future"); return; }`.
- Fix (server): in POST `/api/runs`, reject `new Date(body.date).getTime() < Date.now()` with 400.

### HIGH — No host-unlock gate on server POST `/api/runs`
- File: `server/routes.ts:1150-1162` (POST `/api/runs`)
- What: Server trusts client to enforce host-unlock rules. Someone hitting the API directly (or a bug in the client gate) can host before meeting the participation thresholds described in `replit.md`.
- Fix: in the POST handler, load `user.host_unlocked`; if false, return 403 "Host not unlocked yet."

### HIGH — Location lat/lng 0 accepted; no bounds validation
- File: `app/create-run/index.tsx:239` (client) + `server/routes.ts:1160-1161` (server)
- What: Client guard `!locationLat || !locationLng` treats "0" as falsy but numeric `0` is a valid float. Server accepts any number, including NaN after bad parse.
- Fix (server):
  ```ts
  if (!Number.isFinite(lat) || Math.abs(lat) > 90) return res.status(400).json({ error: "Bad latitude" });
  if (!Number.isFinite(lng) || Math.abs(lng) > 180) return res.status(400).json({ error: "Bad longitude" });
  if (lat === 0 && lng === 0) return res.status(400).json({ error: "Location required" });
  ```

### HIGH — `minDistance == maxDistance` and `minPace == maxPace` accepted
- File: `server/routes.ts:1161` + `app/create-run/index.tsx:324-330` (pace group validation only)
- What: Only `>` is rejected. Equal values produce a zero-width range; downstream code that uses strict `<` or `>` to filter joiners will reject everyone.
- Fix (server): `if (minDist >= maxDist || minPace >= maxPace) return 400`. Mirror client-side before submit.

### HIGH — `maxParticipants` accepts 0, negative, NaN
- File: `app/create-run/index.tsx:275`
- What: `parseInt(maxParticipants)` on `""` returns `NaN`, on "-5" returns -5, etc. Server stores the value. Joining math then always shows "full" or runs forever.
- Fix: `const cap = Math.max(1, Math.min(500, parseInt(maxParticipants) || 20));` on the client, and validate `>= 1` on the server.

### HIGH — Edit endpoint allows scheduling into the past
- File: `server/routes.ts:1689-1709` (PATCH `/api/runs/:id`)
- What: No date validation on edit. Mirrors the creation bug; same ghost-deletion consequence.
- Fix: same `if (new Date(body.date).getTime() < Date.now()) return 400`.

### HIGH — Double-tap on "Post" creates duplicate events
- File: `app/create-run/index.tsx:348-350`
- What: Button is disabled only on `createMutation.isPending`. `onPress` runs synchronously with your validation logic before the mutation's pending state flips on the next tick.
- Fix: add a local `submittingRef` (useRef to bool), set true at top of `validateAndSubmit`, bail out if already true, reset in mutation settled.

### MED — Crew deletion sets `runs.crew_id = NULL`, which leaks crew events into public feed
- File: `server/storage.ts:331` (FK definition) + discover filter in `app/(tabs)/index.tsx`
- What: `ON DELETE SET NULL` preserves the event but drops the crew tie. The discover feed then displays it like any other public event, often with a stale "crew badge" if the client had it cached.
- Fix: change FK to `ON DELETE CASCADE`, or in the deleteCrew transaction, iterate and either cancel-notify participants or convert to host-only events with clear status.

### MED — Cancel-event doesn't clean up `run_tracking_points`
- File: `server/routes.ts:1711-1728` + storage's cancel/delete path
- What: The cascade deletes participants/photos/messages but leaves tracking rows orphaned. Data bloats; if a future "re-open cancelled event" feature exists, old pings will resurrect.
- Fix: add `DELETE FROM run_tracking_points WHERE run_id = $1` to the delete transaction, or set the FK to `ON DELETE CASCADE`.

### MED — Late-start-monitor race on server restart can double-send pre-run pushes
- File: `server/late-start-monitor.ts:220-229` (pre-run already uses `UPDATE ... RETURNING`, good) + the late-start branch which sets the flag *before* sending
- What: For the late-start branch specifically, the flag is written in a separate statement from the push send. Restart between those and the same push goes out twice.
- Fix: use the same atomic-claim pattern: `UPDATE runs SET notif_late_start_sent = true WHERE id = ANY($1) AND notif_late_start_sent = false RETURNING id;` — only push for returned ids.

### MED — Host auto-promotion by GPS silence demotes back-grounded hosts
- File: `server/late-start-monitor.ts:303-348` (`checkAndPromoteHost`)
- What: 5 minutes of no host ping → highest-distance participant becomes host. A phone call or briefly locked phone hits this window easily. After promotion, the demoted host can't "End for Everyone" anymore.
- Fix: lengthen grace to 10 minutes, push a notification to both users on promotion, and allow any original host to resume control within 2 hours.

### LOW — Tz mix in push notification strings
- File: `server/routes.ts:1169-1170` (omits `timeZone`) vs `1215-1216` (forces `"UTC"`)
- What: Two different copies of the same "event at X" string render in different zones. Some users get pushes with a time that disagrees with the app UI.
- Fix: pick one. Recommend always sending ISO in the push payload and formatting in the client with the user's device tz.

## 3) Join / Start / Live participation (`app/run/[id].tsx`, `app/run-live/[id].tsx`, `app/run-spectate/[id].tsx`, server)

### NOT A BUG — Public `/api/runs/:id/live` and `/host-route` are intentional
- Product decision: public runs are watchable live by anyone (including unauthenticated viewers). Solo runs are never watchable. Crew runs aren't listed in discover for non-members, which already handles the visibility there.
- Only fix needed here: if you ever add a true "private, non-crew, non-solo" privacy tier, gate `/live` and `/host-route` behind `run.privacy === "private"` → host + participants only. Until then, no change.

### HIGH — `joinRun` TOCTOU: two simultaneous joiners can exceed `maxParticipants`
- File: `server/storage.ts:1729-1812`
- What: capacity is read in one statement, then checked, then insert in a separate statement. Two concurrent requests both pass the read.
- Fix: single INSERT guarded by a SELECT count in the same transaction (`BEGIN`... `SELECT ... FOR UPDATE` on the `runs` row, check count, INSERT, `COMMIT`). Or:
  ```sql
  INSERT INTO run_participants (run_id, user_id, status)
  SELECT $1, $2, 'joined'
  WHERE (SELECT COUNT(*) FROM run_participants WHERE run_id = $1 AND status = 'joined') < (SELECT max_participants FROM runs WHERE id = $1)
  RETURNING id;
  ```
  Empty result → return "full".

### HIGH — Late joiner never appears until within 500 ft of start
- File: `server/storage.ts:2129-2179` (`pingRunLocation`, is_present gate)
- What: `is_present` is only flipped true when a ping arrives AND user is within 500 ft of the event's start point. A user who joins after the event starts and is already on the route (a mile in) stays invisible on the leaderboard until they back-track into the start zone.
- Fix: on `joinRun`, check `is_active`; if true, immediately query the user's latest ping in the last ~30 s and apply either a wider proximity (route polyline distance not start-point distance) or accept-any-ping-if-already-has-points path.

### HIGH — Ghost-run cleanup can kill legitimately long events
- File: `server/ghost-run-cleanup.ts:5-29` + `server/late-start-monitor.ts:80-196`
- What: Two overlapping cleaners: 4 h no-ping and 8 h total. Marathons, ultras, long group rides with sparse pings are dead in the water.
- Fix: cap "ghost" behavior to runs whose `started_at IS NULL`. For `is_active` runs, require a sliding "no pings from *any* participant in the last 2 h" test, and always notify the host before deleting, with a button to keep the run alive.

### HIGH — `runner-finish` accepts distance=0 / pace=0 and ranks them
- File: `server/routes.ts:1595-1627`, `server/storage.ts:2465-2539` (`finishRunnerRun`)
- What: Zero-distance finishes get a `final_rank`, appear in the leaderboard, and count toward host rating eligibility.
- Fix: in `finishRunnerRun`, if `safeDist <= 0` (or `< 0.1`) and no pings exist, mark as DNF (`final_rank = NULL`, don't affect leaderboard, don't count for host rating).

### MED — Chat after event ended still accepts POSTs
- File: `server/routes.ts:1386-1415` (POST `/api/runs/:id/messages`)
- What: No `is_completed` check; the message is stored but `app/run-live/[id].tsx` stops rendering the chat, so it's dead.
- Fix: reject with 410 if `run.is_completed || !run.is_active`. Alternatively, keep a read-only "Post-run" chat if that's a product decision, and render it.

### MED — Tag-along participants can't chat
- File: `app/run-live/[id].tsx:1069-1130` (`notPresentContainer` block)
- What: `canChatNearby || hasBeenPresent` — tag-alongs are never `is_present`, never within the geofence, so chat is hidden even though they are notionally in the event.
- Fix: also allow chat when the user has a non-null `tag_along_of_user_id` in the participant row.

### MED — `leaveRun` only flips status; leaves tracking points dangling
- File: `server/storage.ts:1829-1834`
- What: Canceled participants' `run_tracking_points` remain. Not broken today, but anything that queries "all tracking points for run X" (host-route, post-run analysis) includes a former participant. If you ever switch from `JOIN participants ON ... WHERE status <> 'cancelled'` to an unfiltered query, it silently corrupts results.
- Fix: either mark tracking points as `is_cancelled = true` on leave, or add a comment asserting the invariant and unit-test it.

### MED — Stale finisher position lingers in live state
- File: `server/storage.ts:2186-2199` (`getLiveRunState`)
- What: After a runner's `/runner-finish`, they stop pinging but `getLiveRunState` LEFT JOINs the latest ping row, so the map shows them stuck at their last position for the rest of the event.
- Fix: prefer `final_distance / finished_at` when `final_pace IS NOT NULL`, and render them as a "Finished" marker distinctly.

### MED — Can join a cancelled / completed event
- File: `server/storage.ts` `joinRun` path + `app/run/[id].tsx` join button visibility
- What: Client hides the join button when `is_completed`, but the endpoint doesn't reject if the client is stale or if the user deep-links.
- Fix: in `joinRun`, `SELECT is_completed, is_cancelled, is_active FROM runs` early, return 409 if not joinable.

### MED — Host starts before scheduled time with no confirm
- File: event detail screen host-start button + server start endpoint
- What: Host can hit Start 2 hours before the announced time. Participants who scheduled their arrival won't see it start. No guardrail.
- Fix: confirm modal if `Date.now() < scheduledDate - 5 min`; server can still allow, but UX should warn.

### LOW — No "event ended" state for a user still on the live screen
- File: `app/run-live/[id].tsx`
- What: When the host finishes, the participant's screen doesn't auto-transition. The live query keeps polling and the user has to manually navigate out.
- Fix: detect `liveState.isActive === false` and push to `/run-results/[id]`.

## 4) Tracking & completion (cross-cutting)

### HIGH — Participant finishing in a live event: distance guard differs from solo
- File: `app/run-tracking.tsx` save path (solo) vs `server/routes.ts:1595-1627` (`/runner-finish`)
- What: The solo save path clamps to 0.001 mi; the event finish path validates differently. A user who finished a joined event with zero movement is recorded differently from the same user in solo mode. Inconsistent data.
- Fix: consolidate a single `minMeaningfulDistance(activity)` helper and use it in both.

### HIGH — App killed mid-event, reopening: tracking doesn't auto-resume in event context
- File: `app/run-tracking.tsx` recovery flow + `contexts/LiveTrackingContext.tsx`
- What: Recovery restores the solo tracking session but doesn't know if you were in an event. User thinks they're still posting pings to the event; they're actually on a new solo session, detached from the event row.
- Fix: include `eventId` in the autosave payload; on restore, re-establish the event tracking context and resume pings.

### MED — `elapsedRef` vs `elapsed` drift in multiple places
- File: `app/run-tracking.tsx` (search all `elapsed` usages)
- What: Same bug class as the solo save issue. Anywhere the code reads `elapsed` (state) instead of `elapsedRef.current` (ref) is a potential 0-1s drift.
- Fix: audit all `elapsed` references in this file and convert to `.current` anywhere outside of JSX render.

### MED — Background task + foreground watcher can both feed `handleCoord`
- File: `lib/locationTasks.ts` + `app/run-tracking.tsx` watch subscription
- What: If background location is granted and the foreground watcher is also running (app on screen), both can deliver coords for the same fix. Distance accumulates twice between same two timestamps.
- Fix: when foregrounded, pause the background task; when backgrounded, stop the foreground watcher. Keep a single source of truth.

## 5) Discover / filter / lists

### HIGH — Completed runs still show "Still joinable" within the 2 h grace window
- File: `app/(tabs)/index.tsx:861-965, 1758-1760`
- What: `matchFuture` hides events > 2 h past their scheduled time, but the "Still joinable" rail doesn't check `is_completed`. A run that legitimately ended at the scheduled time stays on the rail for up to 2 h.
- Fix: in the `isStillJoinableCard` predicate, add `&& !run.is_completed`.

### HIGH — Pace filter is an *overlap* test, not a *fit* test
- File: `app/(tabs)/index.tsx:1724`
- What: `r.min_pace <= applied.paceMax && r.max_pace >= applied.paceMin` returns any event whose range touches the user's selected window. A "6-15 min/mi" event passes the "8-10 min/mi" filter because the ranges intersect.
- Fix: decide product intent. If the user means "I can sustain 8-10 min/mi", the test should be `r.min_pace >= applied.paceMin && r.max_pace <= applied.paceMax` (strict fit), or `applied.paceMin >= r.min_pace && applied.paceMax <= r.max_pace` (user's range fits inside event's). Pick one and document.

### HIGH — New solo run doesn't show up on Discover until refetch timer fires
- File: `app/(tabs)/solo.tsx:688` (save onSuccess) + Discover query
- What: `saveMutation.onSuccess` only invalidates `/api/solo-runs`. Discover's `/api/runs` list isn't touched, so a freshly-saved solo run doesn't appear in the cross-activity feed for up to the refetch interval (15 s).
- Fix: also call `qc.invalidateQueries({ queryKey: ["/api/runs"] })` and, separately, on create-run success in `app/create-run/index.tsx`, invalidate the same key.

### MED — AI search silently returns empty filters when `OPENAI_API_KEY` is missing
- File: `server/ai.ts:12` + `server/routes.ts:890-899`
- What: `generateSearchFilters` returns `null` without logging, route returns 200 with empty filters, client renders empty list indistinguishable from "no matches".
- Fix: return a clear error response (501 + `{ error: "AI search disabled" }`) and fall back to keyword search on the client. At minimum log on the server.

### MED — Filter change doesn't scroll list to top and doesn't reset `eventsExpanded` on crew tab
- Files: `app/(tabs)/index.tsx:1885` (`setVisibleCount(5)`) + `app/(tabs)/crew.tsx:1385-1386`
- What: User at the bottom of a long list applies a filter, and stays visually mid-list while the list underneath is much shorter, making it look empty.
- Fix: on filter change, `flatListRef.current?.scrollToOffset({ offset: 0 })`. On activity toggle in `crew.tsx`, reset `eventsExpanded`.

### MED — Date grouping uses device-local `getDay()` on a UTC timestamp
- File: `app/(tabs)/index.tsx:1741, 1822`
- What: An event at 23:00 local time on the west coast is 06:00 UTC next day; grouping by `new Date(r.date).getDay()` on the server-sent UTC value can slot it under the wrong day in "Today / Tomorrow / This Week" buckets.
- Fix: group by `new Date(r.date).toLocaleDateString(undefined, { weekday: 'long' })` or normalize via `toZonedTime` (if you introduce `date-fns-tz`).

### MED — Pace slider shows `min/mi` label even when distance unit is km
- File: `app/(tabs)/index.tsx:441, 1726` (via `toDisplayPace`)

## 6) Cross-cutting / infrastructure

### MED — ErrorBoundary only catches render errors; most crashes won't reach it
- File: `components/ErrorBoundary.tsx`
- What: No `onError` wired up anywhere in the tree to report; async errors (promise rejections in mutations, GPS callbacks, Speech callbacks) bypass it entirely. Users get a silent broken UI instead of a readable fallback.
- Fix: mount `ErrorBoundary` at `app/_layout.tsx` root. Register a global rejection handler:
  ```ts
  const origHandler = ErrorUtils.getGlobalHandler?.();
  ErrorUtils.setGlobalHandler?.((error, isFatal) => {
    // log to Sentry / your endpoint
    origHandler?.(error, isFatal);
  });
  ```

### MED — `Math.round` on display pace can produce `:60`
- File: `lib/units.ts:37-38`
- What: `const secs = Math.round((pace - mins) * 60);` — when `secs` rounds to 60 (e.g., pace `9.996`), renders as `9:60 /mi`.
- Fix:
  ```ts
  let secs = Math.round((pace - mins) * 60);
  let m = mins;
  if (secs === 60) { secs = 0; m = mins + 1; }
  ```
