# Android Fixes

Five issues surfaced during Android Expo Go testing. Fixes are listed in priority order — the first two are real bugs that will affect all users; #3 is latent; #4 is a theming inconsistency; #5 is likely an Expo Go artifact, not a code bug.

---

## 1. NotificationBanner — Rule of Hooks violation (red-box error)

**File:** `components/NotificationBanner.tsx`
**Severity:** HIGH — causes the red error box on Android.

### Cause

A `useState` hook is declared **after** an early `return null`. When the component first mounts with `bannerQueue.length === 0`, it returns before reaching the `useState` call, so React counts one fewer hook. The moment a banner arrives and the queue becomes non-empty, the early return is skipped and the extra `useState` runs — the hook count changes between renders and React throws:

> Rendered more hooks than during the previous render.

### Current code (abbreviated)

```tsx
export default function NotificationBanner() {
  // ... ~12 hooks declared here (useState, useEffect, useRef, etc.) ...

  // Line ~198
  if (bannerQueue.length === 0) return null;

  // ... JSX / handlers ...

  // Line ~219  ← THIS is the bug
  const [actionPending, setActionPending] = useState(false);

  // ... more JSX using actionPending ...
}
```

### Fix

Move the `useState(false)` declaration up with the rest of the hooks, **before** the early return. Delete the later declaration.

```tsx
export default function NotificationBanner() {
  // ... existing hooks ...
  const [actionPending, setActionPending] = useState(false);  // ← moved to top

  // ... other hooks ...

  if (bannerQueue.length === 0) return null;

  // ... JSX that uses actionPending — unchanged ...
  // DELETE the old `const [actionPending, setActionPending] = useState(false);` line here
}
```

Every hook in a component must be called in the same order on every render. Nothing may precede a hook with a conditional `return`, `if`, `&&`, or loop.

## 2. Run history card — huge blank gap when collapsed

**File:** `app/(tabs)/solo.tsx`
**Severity:** HIGH — visible on every finished run with a route.

### Cause

`<MiniRouteMap>` is rendered **unconditionally** whenever `run.route_path` exists. It is not gated behind the card's expanded state, so every collapsed history row reserves a 200 px map slot, producing the visible empty band.

### Current code — around line 1088

```tsx
{run.route_path && run.route_path.length > 1 && Platform.OS !== "web" && (
  <View style={{ marginTop: 12 }}>
    <MiniRouteMap path={run.route_path} />
  </View>
)}
{isExpanded && (
  <View style={s.statsPanel}>
    {/* ... */}
  </View>
)}
```

### Fix

Add `isExpanded &&` to the map block so the map only mounts when the user opens the card. Bonus: fewer MapViews on the list = much lower memory on Android.

```tsx
{isExpanded && run.route_path && run.route_path.length > 1 && Platform.OS !== "web" && (
  <View style={{ marginTop: 12 }}>
    <MiniRouteMap path={run.route_path} />
  </View>
)}
{isExpanded && (
  <View style={s.statsPanel}>
    {/* ... */}
  </View>
)}
```

## 3. MiniRouteMap — `C` is undefined (latent crash)

**File:** `app/(tabs)/solo.tsx` — lines 100–139
**Severity:** HIGH (latent) — will throw as soon as #2 is fixed and the map actually renders on Android, or if React Compiler ever inlines it.

### Cause

`MiniRouteMap` is a standalone function component. It references `C.primary` on the polyline but **never calls `useTheme()`**. It works today only because the symbol `C` is closed over from an outer scope at runtime — which is fragile, and under the React Compiler experiment (which this app has enabled in `app.json`) that closure can be optimized away, yielding a `ReferenceError: C is not defined` or a stale color.

### Current code

```tsx
function MiniRouteMap({ path, height = 200 }: { path: RoutePoint[]; height?: number }) {
  if (!path || path.length < 2) return null;
  // ... bounding box math ...

  return (
    <MapView /* ...props... */ >
      <Polyline
        coordinates={path}
        strokeColor={C.primary}        // ← C is not defined in this scope
        strokeWidth={3}
        lineCap="round"
        lineJoin="round"
      />
    </MapView>
  );
}
```

### Fix

Import and call `useTheme()` inside the component. While we're here, also remove two props that are invalid/unsupported on Android's MapView under the New Architecture: `userInterfaceStyle="dark"` (iOS-only) and `pointerEvents="none"` (must be a style, not a prop, under Fabric).

```tsx
import { useTheme } from "@/contexts/ThemeContext";

function MiniRouteMap({ path, height = 200 }: { path: RoutePoint[]; height?: number }) {
  const { C } = useTheme();                       // ← added

  if (!path || path.length < 2) return null;
  // ... bounding box math unchanged ...

  return (
    <MapView
      style={{ height, borderRadius: 16, overflow: "hidden" }}
      initialRegion={{ /* ... */ }}
      customMapStyle={MAP_STYLE}
      mapType={MAP_TYPE}
      scrollEnabled={false}
      zoomEnabled={false}
      rotateEnabled={false}
      pitchEnabled={false}
      showsUserLocation={false}
      showsCompass={false}
      showsScale={false}
      showsTraffic={false}
      showsBuildings={false}
      showsPointsOfInterest={false}
      toolbarEnabled={false}
      // removed: userInterfaceStyle="dark"   (iOS-only, ignored / warned on Android)
      // removed: pointerEvents="none"        (not a valid MapView prop under New Arch)
    >
      <Polyline
        coordinates={path}
        strokeColor={C.primary}
        strokeWidth={3}
        lineCap="round"
        lineJoin="round"
      />
    </MapView>
  );
}
```

If the map needs to ignore touches while the parent row is pressable, wrap it in a `View` with `pointerEvents="none"` on the wrapper instead:

```tsx
<View pointerEvents="none" style={{ marginTop: 12 }}>
  <MiniRouteMap path={run.route_path} />
</View>
```

## 4. Share menu stuck in dark theme

**File:** `components/ShareActivityModal.tsx` — lines 64–72
**Severity:** MEDIUM — cosmetic, but a clear theme-mismatch visible to every light-mode user.

### Cause

The modal already imports `useTheme` (line 31) and calls it inside the component (line 95), but all of the chrome uses module-level hardcoded dark constants declared at the top of the file:

```tsx
const PRIMARY = "#00A85E";
const BG = "#050C09";
const SURFACE = "#0D1510";
const CARD_BG = "#0A1410";
const TEXT = "#F0FFF4";
const TEXT_SEC = "#8FAF97";
const TEXT_MUTED = "#4A6957";
const BORDER = "#182B1F";
const GOLD = "#FFB800";
```

Because these are module-scope, they never switch when the user toggles light mode.

### Fix

Replace the hardcoded constants with theme tokens read from `useTheme()`. The component already has the values in local aliases (`themeBg`, `themeSurface`, etc. at lines 105–109) — finish the job by using them everywhere `BG` / `SURFACE` / `TEXT` / etc. are referenced.

**Option A (minimal diff — recommended):** delete the top-level constants and define them from the theme inside the component:

```tsx
// DELETE lines 64–72 (const PRIMARY = ... const BORDER = ...)
// Keep GOLD only if it's genuinely brand-gold rather than a theme token.

export default function ShareActivityModal({ visible, onClose, runData, eventPhotos = [] }: Props) {
  const insets = useSafeAreaInsets();
  const { C } = useTheme();
  const cardRef = useRef<View>(null);

  // Local aliases so the rest of the file can keep its existing names:
  const PRIMARY   = C.primary;
  const BG        = C.bg;
  const SURFACE   = C.surface;
  const CARD_BG   = C.card ?? C.surface;
  const TEXT      = C.text;
  const TEXT_SEC  = C.textSecondary;
  const TEXT_MUTED = C.textMuted;
  const BORDER    = C.border;
  const GOLD      = "#FFB800";   // keep if intentional brand accent

  // ... rest of component unchanged ...
}
```

If any of those references live inside the module-level `StyleSheet.create(...)` block, move those styles inline (or build the stylesheet with `useMemo` inside the component) so they can see the theme values.

**Option B (cleaner):** replace every `BG`/`SURFACE`/`TEXT`/... reference in the JSX with `C.bg`/`C.surface`/`C.text`/... directly and delete the aliases entirely.

Either way, the result is that the share sheet picks up the current theme on every render.

## 5. Tab titles occasionally disappear (Discover / Solo / Crew / Profile)

**File:** `app/(tabs)/_layout.tsx` — line 129
**Severity:** LOW — most likely an Expo Go–only artifact, not a code bug.

### What's happening

`tabBarLabelStyle` applies `fontFamily: "Outfit_600SemiBold"` only when `Platform.OS !== "android"`:

```tsx
tabBarLabelStyle: {
  ...(Platform.OS !== "android" && { fontFamily: "Outfit_600SemiBold" }),
  fontSize: 11,
  marginBottom: Platform.OS === "android" ? 2 : 4,
},
```

So Android falls back to the system font, which rules out missing-font flicker there. What's more likely is that Expo Go on Android hot-reloads the tab bar component before the first paint completes, momentarily rendering labels with zero width until `screenOptions` re-applies.

### What to do

1. **Confirm in a dev build.** Run an EAS dev build (`eas build --profile development --platform android`) and reproduce. If it never happens in the dev build, this is strictly an Expo Go artifact — ignore.
2. **If it reproduces in a dev build**, the most reliable fix is to explicitly pin the label style with a fixed width and ensure `tabBarShowLabel` can't flicker:

   ```tsx
   tabBarShowLabel: true,
   tabBarLabelStyle: {
     fontSize: 11,
     marginBottom: Platform.OS === "android" ? 2 : 4,
     includeFontPadding: false,   // Android-only: prevents layout jitter
   },
   tabBarItemStyle: { paddingTop: 0, paddingBottom: 0 },
   ```

   And make sure `Outfit_600SemiBold` is only applied once the font is loaded — e.g. gate it on the result of `useFonts(...)` in the root layout.

3. Do **not** add this to the priority list unless #1–#4 are shipped and users still report missing titles from a native build.
